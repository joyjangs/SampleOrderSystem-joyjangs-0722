# Phase 7 — 통합 마무리

> 본 문서는 `docs/PLAN.md`의 "Phase 7 — 통합 마무리" 섹션을 구체화한 작업 계획이다. Phase
> 0~6에서 개별적으로 구현/검증된 기능들이 모두 합쳐진 뒤, 시스템 전체를 한 번 더 훑어 완성도를
> 확인하는 단계다.

## 0. Phase 7의 성격 — 다른 Phase/에이전트와의 역할 구분

- Phase 0~6 각각은 기능 구현 → `code-review` → `tester` → (직후) `supervisor` 순으로 이미
  검증되었다는 것을 전제로 한다. Phase 7은 이 개별 검증들을 반복하는 것이 아니라, **모든
  Phase가 합쳐진 최종 결과물 전체**를 대상으로 한 번 더 훑는 마무리 게이트다.
- CLAUDE.md에 정의된 `supervisor`는 "매 에이전트 호출 직후" 바로 요약을 제공하는 역할이다.
  Phase 7은 그 반복적 단계별 확인과 겹치지 않도록, **전체 Phase가 끝난 뒤 1회** 수행하는
  종합 점검(전체 시스템 테스트 재실행, 전체 diff 최종 code-review, 문서-구현 정합성 확인,
  커밋 이력 정리)으로 범위를 한정한다.
- Phase 7 자체에서도 `code-review`/`tester`를 호출하지만, 이는 "전체 diff"와 "전체 시스템
  테스트"를 대상으로 하는 최종 1회성 점검이며, 끝난 뒤 각각 `supervisor`로 재확인을 받는
  흐름은 CLAUDE.md의 기본 흐름을 그대로 따른다.

## 1. 목표 요약 (PLAN.md 원문 인용)

**목표**: 전체 흐름 검증 및 문서 정리.

- 전체 시스템 테스트(시료 등록 → 주문 → 승인/생산 → 출고 → 모니터링) 재실행
- `code-review` 에이전트로 전체 diff 최종 점검
- `CLAUDE.md`/`docs/PRD.md`에 구현 중 확정된 세부사항 반영 여부 확인
- 커밋 이력 정리 (PRD 9.3 "의미 있는 Commit 이력 관리")

**완료 기준**: 전체 빌드/테스트 통과, 문서와 실제 구현 간 불일치 없음.

### 2.2 code-review 최종 점검 (전체 diff 대상)

- [ ] `git log`/`git diff` 등으로 프로젝트 시작 시점부터 현재까지의 전체 변경사항을 파악

      교차 Phase 이슈를 점검:

        않았는지)

        하드코딩되어 OCP를 깨는지)
- [ ] 리뷰 결과에 대해 직후 `supervisor`를 호출해 검증받는다 (CLAUDE.md 기본 흐름 준수)

## 4. 의존 관계

- **전제(입력)**: Phase 0~6이 각각 `code-review`/`tester`/`supervisor` 검증을 통과해 병합된
  상태. Phase 7은 이 산출물들이 실제로 존재한다는 것을 전제로 하며, 만약 특정 Phase가 아직
  완료되지 않았다면 Phase 7을 시작할 수 없다(선행 조건 미충족 시 사용자에게 먼저 보고).
- **산출(출력)**: 이 프로젝트의 최종 단계이므로 이후 Phase는 없다. Phase 7의 산출물은
  "빌드/테스트 통과 + 문서-구현 정합성 확인 완료 + 정리된 커밋 이력"이라는 최종 상태 그
  자체이다.

## 5. 핵심 비즈니스 로직 주의사항 (회귀 검증 대상)

Phase 7에서 새로 도입되는 로직은 없지만, 통합 시점에 가장 깨지기 쉬운 핵심 규칙이므로
아래 내용을 최종 e2e/회귀 테스트에서 반드시 재확인한다 (CLAUDE.md, PRD 7.4 원문 인용).

- **승인 시점 생산량 확정 규칙**: 실 생산량/생산 시간 계산은 승인 시점의 재고만 고려해 1회
  확정하며, 생산 큐 대기 후 실제 생산 시작 시 재고를 재계산하지 않는다. 승인은 주문이 들어온
  순서대로 재고를 소진하며 계산해야 하고, 물리적 `stock`은 실제 생산 완료/출고 시점에만
  변경한다. 모니터링에는 항상 물리적 `stock`을 표시한다.

  ```
  예시1) 현재 재고 0, A시료 수율 0.5
    주문1: A 50개 주문 → 부족분 50 → 실 생산량 = ceil(50/0.5) = 100
    주문2: 주문1 생산 중 A 50개 추가 주문 → 승인 시점 재고는 여전히 0 →
           부족분 50 → 실 생산량 = ceil(50/0.5) = 100 (주문1의 미래 생산량은 고려하지 않음)

  예시2) 현재 재고 90, A시료 수율 1.0
    주문1: A 100개 주문 → 부족분 = 100 - 90 = 10 → 실 생산량 10
    주문2: A 10개 주문 → 물리적 재고 90개는 이미 주문1의 출고분으로 예정 → 가용 재고 0 →
           실 생산량 10 (모니터링에는 물리적 재고 90개 그대로 표시)
  ```

- 이 두 예시가 Phase 3 테스트에 이미 있는지 확인하고, 없다면(혹은 리팩터링 중 삭제/약화된
  경우) Phase 7에서 회귀 테스트로 반드시 복원한다.
- `REJECTED`는 모니터링 집계에서 제외되어야 한다 — 전체 e2e에서 거절 케이스를 하나 포함해
  모니터링 화면에 반영되지 않는지 확인한다.

## 6. 테스트 계획 (`tester` 에이전트, 최종 1회성 시스템 테스트)

- **전체 통합 시나리오 (System Test)**
  - 시료 2개 이상 등록 → 각각에 대해 여러 건 주문 → 일부는 거절, 일부는 승인(재고 충분/
    부족 혼합) → 생산 큐 FIFO 순서 진행 → 생산 완료 후 `CONFIRMED` → 출고 → `RELEASE`까지
    한 번의 테스트 흐름으로 검증
  - 모니터링 결과(상태별 건수, 재고 여유/부족/고갈)가 위 시나리오 종료 시점의 실제 데이터와
    일치하는지 검증
- **영속성 회귀 테스트**
  - 위 시나리오 중간(예: `PRODUCING` 상태 존재 시점)에 애플리케이션을 재시작(파일 재로드)
    한 뒤에도 주문/생산 큐 상태가 그대로 복원되는지 검증
- **PRD 7.4 예시 1·2 회귀 테스트**
  - Phase 3에서 작성된 케이스가 최종 빌드에서도 여전히 통과하는지 재실행으로 확인 (내용은
    본 문서 5절 참고)
- **더미 데이터 통합 회귀 테스트**
  - 더미 데이터 생성 기능이 실제 Repository와 통합되어, 생성 후 메인 메뉴 요약/모니터링에
    정확히 반영되는지 확인
- 테스트 결과는 통과/실패 개수와 실패 원인을 명확히 보고하며, 실패를 감추기 위해 프로덕션
  코드나 테스트 기대값을 몰래 조정하지 않는다.
